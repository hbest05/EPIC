Message encryption:
AES-256-GCM for message encryption (via Double Ratchet message keys)
- AES-256 gives 256-bit key length; GCM gives authenticated encryption with 128-bit tag (forgery probability 2⁻¹²⁸ per attempt)
- Nonces managed internally by the Double Ratchet — one fresh message key per message, used once then erased, so nonce reuse is structurally impossible

Associated data (attached to every AEAD call):
IK_A || IK_B — binds ciphertext to this specific session (prevents unknown key-share attack)
Ratchet public key — binds ciphertext to the current DH epoch
PN (previous chain length) + N (message index) — prevents reorder/replay attacks on the chain
(Note: HPKE enc blob from old design dropped — X3DH + Double Ratchet header serves the same binding purpose)

Key exchange — how 2 parties agree on shared secret:
X3DH (Extended Triple Diffie-Hellman) — Signal specification
- Bob prepublishes a key bundle to the server (IK, SPK, OPKs — see below)
- Alice fetches the bundle and runs 4 DH operations asynchronously (Bob can be offline)
- SK = HKDF-SHA256(DH1 || DH2 || DH3 || DH4) seeds the Double Ratchet root key
- Implemented via libsignal (Python: python-libsignal bindings; C++: libsignal-protocol-c)

Per-message key evolution — Double Ratchet Algorithm:
- Symmetric ratchet: (CK_next, MK) = HKDF(CK_current) — advances per message, erases MK after use → forward secrecy
- DH ratchet: fresh X25519 keypair injected every round-trip, re-seeds root key → post-compromise security (heals after 1 round trip)
- Net result: compromise of one message key or full ratchet state does not expose past messages; future messages recover after one undisturbed reply

Elliptic curve:
X25519 (Curve25519) — used throughout X3DH and Double Ratchet DH steps

Key bundle published per user (X3DH):
Identity Key (IK) — long-term X25519 keypair + Ed25519 signing key, never rotates
Signed Prekey (SPK) — X25519 keypair signed by Ed25519 IK, rotated ~weekly
One-Time Prekeys (OPKs) — pool of ~100 X25519 keypairs, each used exactly once then deleted
- OPK depletion fallback: X3DH drops to 3 DH ops (DH4 omitted), slightly weaker FS vs SPK leak

Password hashing server side:
Argon2id — memory: 64 MiB, iterations: 3, parallelism: 4 (~500ms on target hardware)

Local client private key protection at rest:
HKDF-SHA256(user_passphrase, salt_local, info="cs4455-local-key-enc-v1") → AES-256-GCM(sk)
- Separate KDF parameters from server-side Argon2id intentionally — different threat model (stolen device vs server breach)

Trust model between two parties:
TOFU + SHA-256 fingerprint pinning over IK bytes
- Server compromise can MitM first contact (before pinning) — this limitation is acknowledged explicitly
- After pinning, a compromised server cannot substitute keys for existing contacts

Sender auth:
X3DH DH1 = DH(IK_A, SPK_B) — mutual authentication baked into the handshake
Ed25519 signature on SPK by IK — Alice verifies the bundle came from Bob, not the server
- Deniable: DH-based auth means Bob cannot prove to a third party that Alice sent a specific message (no transcript signatures)

Internal hash:
HKDF-SHA256 — used for SK derivation in X3DH, chain/message key derivation in Double Ratchet, and local key derivation
